## Backend Development
- Import Zod schemas from `@canopy/shared` for input validation — do NOT redefine types in backend
- Validate request bodies with `.parse()` and return responses that satisfy shared response schemas
- Write Lambda handlers in TypeScript with proper input validation (zod)
- Use the AWS SDK v3 (@aws-sdk/client-dynamodb, @aws-sdk/lib-dynamodb)
- **Use `crypto.randomUUID()` for generating IDs** — it's built into Node.js 20. Do NOT use the `uuid` npm package — it causes esbuild bundling failures in CI/CD.
- Design DynamoDB tables with single-table design patterns
- Always handle errors gracefully and return appropriate HTTP status codes
- Use environment variables for table names and configuration (never hardcode)

### Resource Naming Convention
All AWS resources MUST be prefixed with `canopy-`. This is required because the agent's IAM permissions are scoped to `canopy-*` resources only. If you name a resource without this prefix, you will not be able to read from it to verify tests.

Examples:
- DynamoDB tables: `canopy-projects-table`, `canopy-users-table`
- Lambda functions: `canopy-create-project`, `canopy-list-projects`
- CloudFormation stack: `canopy-app-stack`

CloudWatch log groups automatically follow `/aws/lambda/canopy-*` if Lambda functions are prefixed correctly.

**If you name a table `ProjectsTable` instead of `canopy-projects-table`, `aws dynamodb scan` will be blocked by IAM.**

## Infrastructure as Code
- Write CDK stacks in TypeScript using aws-cdk-lib v2
- **Use `NodejsFunction` from `aws-cdk-lib/aws-lambda-nodejs`** for Lambda handlers — it bundles with esbuild automatically, resolving imports from `@canopy/shared` at synth time. Do NOT use `lambda.Function` with `Code.fromAsset()` — that requires pre-built artifacts and breaks CI/CD.
- Configure `NodejsFunction` bundling to only externalize AWS SDK: `bundling: { externalModules: ['@aws-sdk/*'] }`. All other dependencies (zod, etc.) must be bundled by esbuild, not externalized. Avoid npm packages that aren't needed — prefer Node.js built-ins (e.g., `crypto.randomUUID()` instead of `uuid`).
- In `cdk.json`, use `tsx` instead of `ts-node`: `"app": "npx tsx bin/canopy.ts"`. Add `tsx` to devDependencies. Do NOT use `ts-node` — it has path resolution issues in CI/CD.
- Follow least-privilege IAM patterns
- Write CDK tests (snapshot + fine-grained assertions)
- Run `cdk synth` to validate templates locally before committing
- Never run `cdk deploy` — deployment happens via CI/CD
- Stack outputs must include: ApiUrl, FrontendBucketName, DistributionId
- Use CfnOutput for all values that other systems need

### Required CDK Construct IDs (DO NOT CHANGE)
A CloudFormation stack (`canopy-app-stack`) already exists from a previous deployment. You MUST use these exact construct IDs so CDK generates matching logical IDs and updates the existing stack instead of trying to create duplicate resources. Changing any construct ID will cause "already exists" errors during deployment.

| Resource | Stack ID | Construct ID | Physical Name |
|----------|----------|-------------|---------------|
| DynamoDB Table | `canopy-app-stack` | `CanopyMainTable` | `canopy-main-table` |
| Lambda Function | | `CanopyApiHandler` | `canopy-api-handler` |
| HTTP API | | `CanopyHttpApi` | `canopy-api` |
| Lambda Integration | | `CanopyLambdaIntegration` | (auto) |
| S3 Bucket | | `CanopyFrontendBucket` | `canopy-frontend-{account}-{region}` |
| CloudFront Distribution | | `CanopyDistribution` | (auto) |
| Origin Access Identity | | `OAI` | (auto) |

**CDK app entry point** (`bin/canopy.ts`): `new CanopyStack(app, 'canopy-app-stack', ...)`

**API Gateway routes**: Use separate per-method routes (`GET`, `POST`, `PUT`, `DELETE` on `/{proxy+}`) plus a `GET` on `/`. Do NOT use `ANY /{proxy+}` — the existing stack has per-method routes and switching to ANY changes the logical IDs, causing deployment conflicts.
